# Phenology Observation Portal (POP) Retrofitting

This document describes the requirements to modernize the NPN's Phenology Observation Portal (POP) interface to communicate asyncronously with serverless/Tinybird cloud configuration rather than communicating directly with self-hosted services and database, and attempting to do so syncronously. 

The scope of this project encomposes first retrofitting the Status and Intensity type download, and will be expanded in the future to include the other download types.

This project was also already partially converted and contains some incomplete proof of concept work from the last few commits.

## Components

### Tinybird/Clickhouse

There's now a tinybird/Clickhouse database handled through a series of serverless Lambda functions on AWS. The structure of this database is largely abstracted away from this front end that only concerned itself with getting the data through the lambda functions.


### Lambda Function

#### Creating Async Data Request

This is the endpoint for receiving requests from this interface, replacing the existing endpoint.

URL: https://5cpb3wyvskklwikuxndqilxsae0elgst.lambda-url.us-west-2.on.aws/

The contract for this function is essentially the same as the existing endpoint; paramater type/naming conversion happens serverside. The only exception is are the Optional Fields. This field has been replaced with a series of grouped flags as follows:

include_submission (String)
include_observation_detail (String)
include_species_detail (String)
include_site_detail (String)
include_individual_detail (String)
include_phenophase_detail (String)
include_climate (String)
include_remote_sensing (String)

They all take 1 as input if desired or ommitted if not requested.

This endpoint in the happy path returns a 202 response including a job_id parameter in the json body.


#### Check Async Status Request

This is an endpoint for checking the status of the submitted job. The job_ids from the create request can be submitted as a GET paramater (jobId). This will return the status of the request; "complete" on a finished job "processing" on a pending request, or "failed" on a failure.

URL: 

On a completed response, it will also return a presigned S3 URL "download_url", that can be used to download the data from S3 directly.


#### Observation Count Endpoint

It's not built yet, but we will need to add a lambda function or direct connection to tinybird for requeting the count of observations generated by the request if the user requested data based on the currently selected filter options. We can scaffold for this but cannot build it in full for now. Mock requests can just return 50M.


### Workflow

This front end should be updating the count of observations whenever the user changes the set of filters, i.e. as it functions today, but using the new endpoint.

But more importantly, the download function should now submit the request. This client should then poll the status endpoint first after 5 seconds and then every 10 seconds after that. When the request comes back as "complete" then it should use the presigned URL to download the data.

### UI Changes

There's one major UI change that needs to happen. The Output Fields screen needs to be re-worked significantly. Currently there's a set of usually three tabs. We will reduce this to two "Optional" and "Default". The default section will remain largely unchanged, but the optional section will have a series of categories grouped into checkboxes rather than each individual field being available for selection. Under each grouping the included fields should be listed with the tool tips still available. The groups are mapped as follows:

### Coding Standards

This is a legacy project. We want to touch as few lines as code as possible. We want the changes to be surgical and non-invasive. We generally want to follow good coding practices as much as possible but still follow the existing conventions of the project whenever possible. There's a lot of work here that is out of scope.
